## 背景和动机
### 背景
LLM-based generative recommendation 的基本流程是：把用户历史交互编码成 token 序列，再让 LLM 生成下一个 item。其中最关键的设计之一是 **item identifier**，也就是“一个 item 应该如何映射成 LLM 可读可生成的表示”。现有 identifier 大致分两类：
1. **Token-sequence identifier**：把 item 表示成多个离散 token 的序列。
2. **Single-token identifier**：把 item 表示成单个 embedding（如 ID embedding 或 semantic embedding）。

但无论是哪种设计，都存在一些问题亟待解决。**Token-sequence identifier**有如下两个问题：
- **Local Optima in Beam Search**：生成是逐 token 自回归进行的，beam search 每一步都只保留局部 top-K 前缀；如果目标 item 的前几个 token 概率不高，就会被过早剪枝，参见图一。
- **生成效率低**：生成一个 item 需要多步串行 LLM 调用，推理延迟和算力成本都很高。

**Single-token identifier** 也同样有明显短板：
- 如果用 **ID embedding / CF embedding**，会高度依赖交互数据，对长尾 item 和 cold-start 场景不友好。
- 如果只用 **semantic embedding**，又会缺失推荐里很关键的 Collaborative Filtering 信息，个性化能力不足。
### 动机
作者认为忽略 identifier中 token 之间依赖性有助于并行的 token 生成，同时提高推理效率。基于此，他们为 LLM-based generative recommendation 提出一种新颖的 set idenfier 范式，如图2所示。

这个设计应满足如下两个原则：
1. **同时整合 semantic + CF information**：兼顾泛化能力和个性化能力。
2. **Order-agnostic identifier**：item 内不同维度的信息不应被强行串成有序序列，否则会引入不必要的 token dependency，并诱发 beam search 的局部最优问题。
## 方法总览
SETRec 的核心思想是不要把一个 item 表示成一个 token，也不要表示成有顺序的 token 序列，而是表示成一个“无序 token 集合”。每个 item 的 set identifier 由两部分组成：
- **一个 CF token**：表示用户行为交互中学到的协同过滤信息。
- **多个 semantic tokens**：表示 item 语义的不同潜在维度，而不是把所有语义压成一个 embedding。

SETRec整体流程可以分为两步：
1. **Order-agnostic Item Tokenization**
	- 用 CF tokenizer 提取 item 的 CF embedding。
	- 用 semantic tokenizer 把 item 语义编码成多个 semantic embeddings。
	- 两者合起来构成 item 的 set identifier。
2. **Simultaneous Item Generation**
	- 用户历史中的每个 item 都被展开为一个 token set。
	- 通过稀疏注意力去掉同一个 item 内 token 之间的依赖。
	- 用不同的 query vectors 同时生成 CF / S1 / S2 / ... / SN 各个维度的 token。
	- 再通过 grounding head 把生成的 token set 映射回真实 item。

SETRec 的模型架构如图3所示，包括 order-agnostic item tokenization 和 simultaneous item generation 两个模块。下面分别介绍这两个模块的细节。
### Order-agnostic Item Tokenization
**CF Tokenizer**：作者用 SASRec 和一个线性投影层来获得 item CF mebedding $z_\text{CF} \in \mathbb{R}^d$。

**Semantic Tokenizer**：作者先用预训练的 semantic extractor（Sentence T5）提取语义信息，然后通过训练一个 AE 来获取  semantic token。AE 的 encoder 把 item 语义编码成一个 latent vector：
$$
z = [z_{S_1}, z_{S_2}, ..., z_{S_N}] \in \mathbb{R}^{Nd}
$$
decoder 则把 $z_\text{sem}$ 解码成 N 个 semantic tokens $\{s_1, s_2, ..., s_N\}$。semantic tokenizer 采用如下重构损失进行训练：
$$
\mathcal{L}_\text{AE} = \| s - \hat{s} \|_2^2
$$
这里$\hat{s} = \text{Decoder}(z)$是 decoder 的输出，$s$是 extractor 的输出。

**Token Corpus**：通过上述两个 tokenizer，我们就得到了 CF token 和 N 个 semantic tokens。把它们合起来就构成了 item 的 set identifier，即$\hat{i} = {z_\text{CF}, z_{S_1}, z_{S_2}, ..., z_{S_N}}$。因此，我们可以为每一个信息维度准备一个 token corpus，$\mathcal{Z}_\text{CF}, \mathcal{Z}_{S_1}, \ldots, \mathcal{Z}_{S_N}$。这些收集到的 token corpus 会用作 grounding head，以便更有效地将生成出的 token 映射回真实 item。（详细内容参见 Query-guided Generation）
### Simultaneous Item Generation
#### Query-guided Generation
如图3所示，为了让模型生成的 token 对齐多个信息维度，作者引入了一组可学习的 query vectors $q \in \mathbb{R}^d$，$d$是模型隐藏层维度，让模型更好区分不同信息维度分别代表什么。形式上，每个信息维度的 generated token $\hat{z}_k$由如下公式给出：
$$
\begin{cases}
\mathbf{x} = [\{\mathbb{z}_{CF}, \mathbb{z}_{S1}, \dots, \mathbb{z}_{SN}\}_1, \dots, \{\mathbb{z}_{CF}, \mathbb{z}_{S1}, \dots, \mathbb{z}_{SN}\}_L]
\\
\hat{\mathbf{z}}_k = \text{LLM\_Layers}(\mathbf{x}, \mathbf{q}_k)
\end{cases}
$$
这里 $q_k$ 是第 $k$ 个信息维度可学习的 query vector。通过上式，我们可以获得所有信息维度的 generated token，并且得到最终生成的 set identifier$\hat{i} = \{\hat{z}_\text{CF}, \hat{z}_{S_1}, \ldots, \hat{z}_{S_N}\}$。

**Token Generation Optimization**：为了实现准确的 item 推荐，我们希望 generated token 尽可能和 target token 在每个维度对齐：
$$
\mathcal{L}_\text{Gen} = -\frac{1}{|\mathcal{D}|}\sum_{\mathcal{D}}\sum_{k \in \mathcal{F}}\frac{\exp(sim(\hat{z}_k, z_k))}{\sum_{z' \in \mathcal{Z}_k} \exp(sim(\hat{z}_k, z'))}
$$
这里$\mathcal{F} = \{\text{CF}, S_1, \ldots, S_N\}$，$sim(\cdot)$是相似度函数，$\mathcal{Z}_k$是第 $k$ 个信息维度的 token corpus。
**Token Generation Grounding**：为了解决生成的 tokens 组合远大于 item corpus 的问题，作者设计了一个策略，把每个维度生成的 token 分别对所有 item 打分，再做线性融合：
$$
\begin{cases}
s_k = W_k\hat{z}_k
\\
s = (1 - \beta)s_{\text{CF}} + \beta \sum_{k \in \mathcal{F}/\ \text{CF}} s_k
\end{cases}
$$
其中，$W_k \in \mathbb{R}^{|\mathcal{Z}_k| \times d}$从 $\mathcal{Z}_k$ 中得到，$\beta$是一个超参数，用来平衡语义维度和协同信号维度的权重。

### Instanitiation
为了在 LLMs 上实例化 SETRec，作者通过最小化如下损失来优化 CF，semantic tokenizers 和 learnable query vectors：
$$
\mathcal{L} = \mathcal{L}_\text{GEN} + \lambda \mathcal{L}_\text{AE}
$$
其中 $\lambda$ 是一个超参数，用来平衡生成损失和 AE 损失。在推理时， SETRec 首先将所有 items 分词成 set identifiers 并得到 token corpus $\mathcal{Z}$。推荐阶段，SETRec 将 user history 转换成 identifier sequence，通过稀疏注意力掩码进行 query-guided simultaneous generation，得到每个信息维度的 tokens。最后，通过 grounding head 把生成的 token 映射回 item corpus，得到最终推荐结果。
## Experiment
作者做了许多实验，旨在回答如下4个问题：
- **RQ1**：在不同类型的 LLM 架构上，SETRec 相比各类 item identifier 基线方法的效果如何？
-  **RQ2**：SETRec 的不同组件（如 CF token, semantic token, query-guided generation, sparse attention）对性能的贡献如何？
- **RQ3**：随着模型参数规模增大，SETRec 的性能表现如何？它提升整体推荐效果的机制是什么？
- **RQ4**：semantic embeddings 的数量、tokenizer 训练强度以及推理阶段的 semantic strength 对 SETRec 性能有何影响？
### Experimental Settings
#### Datasets
作者采用三个 Amazon review datasets 上流行的数据集：Toys，Beauty和Sports，以及 steam 数据集。评估指标使用Recall@K 和 NDCG@K（K=5，10）。
#### Baselines
作者将 SETRec 和以下 baselines 进行对比，包括 single-token identifiers 和 token-sequence identifiers 两大类：
- **DreamRec**：一种相关性很高的方法，使用 ID embedding 表示每个 item，并采用扩散模型对 LLM 生成的 ID embedding 进行细化。
- **E4SRec**：利用预训练的 CF 模型获得 ID embedding，并使用线性投影层高效地计算 item 分数。
- **BIGRec**：采用 item 标题作为 identifier，其中 token 来自人类词表。
- **IDGenRec**：一种可学习的 ID 生成器，目标是从人类词表中生成简洁但信息丰富的标签来表示每个 item。
- **CID**：利用层次聚类得到 token 序列，并通过 item 共现矩阵构建 identifiers，从而让交互行为相似的 items 共享相似 tokens。
- **SemID**：使用外部 token 序列表示 items，该序列基于 item 的层次类别信息构建。
- **Tiger**：采用带 codebook 的 RQ-VAE，将 item 语义信息量化为由外部 tokens 组成的 token 序列；其 identifier 按从粗粒度到细粒度的顺序组织信息。
- **LETTER**：在 RQ-VAE 训练中同时引入语义信息与 CF 信息，得到具有多维信息且多样性更好的 identifiers。
#### Implementation Details
作者将方法应用在两种不同架构的 LLMs 上，分别是 encoder-decoder 架构的 T5-small和 decoder-only 架构的 Qwen2.5。对于 LLM 的训练，所有方法训练都采用相同的 prompt，"What would the user be likely to purchase next after buying items in their history?"。
### Overall Performance (RQ1)
